逆向分析


打开了DEP、GS、dynamicbase、SEH

程序比较简单，有三次栈溢出，并通过puts打印到屏幕上

int __fastcall main(int argc, const char **argv, const char **envp)
{
  FILE *v3; // rax
  FILE *v4; // rax
  FILE *v5; // rax
  int v6; // ebx
  char DstBuf[256]; // [rsp+20h] [rbp-118h] BYREF

  v3 = _acrt_iob_func(0);
  setbuf(v3, 0i64);
  v4 = _acrt_iob_func(1u);
  setbuf(v4, 0i64);
  v5 = _acrt_iob_func(2u);
  setbuf(v5, 0ui64);
  v6 = 3;
  do
  {
    --v6;
    memset(DstBuf, 0, sizeof(DstBuf));
    puts("input:");
    read(0, DstBuf, 0x400u);
    puts("buffer:");
    puts(DstBuf);
  }
  while ( v6 > 0 );
  return 0;
}

dll 版本是 Windows Server 2019

控制了循环，第四次走system getshell…果真如此吗？

对StackCookie3的校验失败：

这是由于通过 rbx 控制循环导致了 new_rsp 与泄露 StackCookie3 时的 old_rsp 不同，StackCookie3 ^ new_rsp != __security_cookie

既然要走ret2dll，rsp的变动是无法避免的，要得到新的 StackCookie 就需要泄露栈值和__security_cookie：我们可以 ret2dll 泄露__security_cookie，再与 StackCookie3 异或得到 old_rsp

  • 第三轮正确操作
    第一次继续连带泄露 StackCookie3
    第二次paddings
    第三次使用 ret2dll 泄露__security_cookie，并通过 rbx 控制循环，通过异或得到 old_rsp，从而计算出任意rsp的StackCookie
    第四次 ret2dll 泄露 ucrtbase，并通过 rbx 控制循环
    第五次走 system getshell
